Hooks are scripts that GIT runs automatically at certain points in its workflow. They let you enforce policies, validate input, or trigger actions around events like committing, merging, or pushing. Every repository ships with sample hooks in the .git/hooks directory (named *.sample); remove the suffix and make the file executable to enable one.

Because hooks live inside .git/, they are local to a clone and are not committed or distributed with the repository for security reasons. That is exactly the gap the Husky article solves.

GIT also has a set of server-side hooks (such as pre-receive, update, and post-receive) that run on the remote when it receives a push. They are used to enforce policies centrally or trigger deployments and CI, but they live on the server rather than in your local clone, so they are out of scope here. You can find the full list, along with the exact arguments each client-side and server-side hook receives, in the official reference: https://git-scm.com/docs/githooks.
