What a GIT Worktree Is

A git worktree is "a separate working directory that shares the same .git history and remote, but sits on its own branch to avoid interruptions."

Key characteristics:

  • One branch, one worktree. Git prevents checking out identical branches across multiple worktrees simultaneously, preventing accidental conflicts.
  • It's a fresh checkout. Dependencies and environment files must be reinstalled in the new directory.
  • They stay linked. Worktrees remain connected through git tracking, allowing operations like rebasing to function normally.

Core Git Commands

# create a worktree on a NEW branch, in a sibling folder
git worktree add ../repo-hotfix -b hotfix

# ...or check out an EXISTING branch (omit -b)
git worktree add ../repo-hotfix hotfix

git worktree list          # show every worktree and the branch each holds
git worktree remove ../repo-hotfix   # delete when done (cleans up dir + metadata)

Manually deleting worktree folders leaves stale metadata. Use git worktree remove instead, which handles cleanup completely.

Claude Code's Built-in Worktrees

claude --worktree feature-auth   # or -w; start Claude in an isolated worktree
claude --worktree                # omit the name → auto-named, e.g. bright-running-fox
claude --worktree "#1234"      # branch from a GitHub PR

Features:

  • Worktrees are created under .claude/worktrees/<name>/ on a new branch worktree-<name>
  • Add .claude/worktrees/ to .gitignore
  • Mid-session creation via the EnterWorktree tool
  • Use .worktreeinclude to copy gitignored files into worktrees
  • Set isolation: worktree in subagent frontmatter for parallel work
  • Clean worktrees auto-remove on exit; dirty ones prompt for removal

Set worktree.baseRef to "head" in settings to start from unpushed local work.
